# HoneyComb (허니콤) — 프로젝트 스펙

## 프로젝트 개요

- **프로젝트명**: HoneyComb (허니콤) — V2G 충·방전 스케줄링 플랫폼
- **차주용 앱 이름**: 꿀차지 (Honey Charge)
- **목표**: 제주·호남의 재생에너지 출력제어(버려지는 전력) 문제를, 신규 인프라 투자 없이 운행 중인 전기차 배터리를 V2G로 활용해 완화한다. 발전량·수요·유휴시간을 AI로 예측하고 MILP로 충·방전 스케줄을 최적화하며, "어디에·언제·몇 대 규모로 확대해야 경제성이 나오는지"를 정부가 판단할 수 있는 근거를 제공한다.
- **산출물**: ① 정부용 정책 웹 대시보드 ② 차주용 모바일 앱(꿀차지)
- **성격**: 실차 제어 하드웨어가 아닌 **데이터 기반 예측·최적화·의사결정 지원 도구**. 해커톤 데모 수준. 모든 무거운 연산은 사전 계산, 현장에서는 정적 데이터 재생.
- **브랜드**: 메인 컬러 `#ffcd4a`(노란색), 벌집/꿀 메타포. 대시보드 지도는 보로노이 셀 → 육각 벌집 연상.

## 핵심 설계 원칙

1. **사전 계산 → 정적 파일 재생**: 예측·최적화·입지분석 결과는 전부 JSON/GeoJSON 정적 파일. 웹/앱은 뷰어. 서버 없이도 시연 가능해야 함.
2. **QGIS는 분석 도구, 웹은 렌더러**: 입지 분석은 QGIS에서 완료 → GeoJSON(EPSG:4326) export → Leaflet 렌더링. 웹에서 공간 연산 금지.
3. **생성형 AI(Claude API)가 핵심 기능**: 대시보드의 정책 브리핑 자동 생성. 심사 기술성 요건이므로 장식이 아닌 실제 동작 기능이어야 함.
4. **동적 재계산은 수식만**: EV 대수 슬라이더는 사전 계산된 시나리오 데이터셋을 보간/재계산(JS)하는 것이지, 최적화를 다시 돌리지 않음.

## 시스템 구조 (데이터 흐름)

```
[데이터 소스]
  KPX 출력제어량(시간별) / 기상(일사량·풍속) / 전력수요  → 전처리 파이프라인
  차량 대여·유휴 패턴(실증+시뮬레이션)                  → 예측 모델
  충전소 위치·상권정보·EV 등록/통행량                   → QGIS 입지 분석

[사전 계산 — 해커톤 이전 완료]
  A. 예측: LightGBM/XGBoost로 발전량·수요·유휴시간 예측
     → MAPE·R²로 모델 비교, 최적 모델 선정
  B. 최적화: MILP 충·방전 스케줄 (목표 충전량 제약 + 배터리 열화비용 목적함수 반영)
  C. 입지: 보로노이 셀 → 도보권 버퍼 교집합(쿠폰 낙수 범위)
     → 적합도 점수(잉여량·상권밀도·접근성, 컬럼 분리 저장) → GeoJSON export

[산출 정적 데이터]
  scenarios.json  : EV 대수별 절감액·회피량·스케줄 시나리오
  siting.geojson  : 보로노이 셀 + 적합도 점수(구성요소별 컬럼)
  schedule.json   : 잉여전력 곡선 + 충·방전 스케줄 시계열

[산출물 1: 웹 대시보드 (정부용)]
  W1 입지 지도 — Leaflet + GeoJSON, 셀 클릭 시 점수 분해 팝업
  W2 스케줄 뷰 — 잉여전력 곡선 위 충·방전 스케줄 차트
  W3 EV 슬라이더 — 대수 조정 → 절감액·회피량 실시간 재계산
  W4 시나리오 비교 뷰 — 정책 시나리오별 비용/편익
  W5 Claude API 정책 브리핑 — W3·W4의 현재 상태를 입력으로 브리핑 텍스트 생성

[산출물 2: 차주용 앱 꿀차지]
  M1 최적 참여시간 + 예상 수익 안내
  M2 누적 수익 대시보드 — 방전량·정산액·배터리 기여도
  M3 꿀 적립·지역 쿠폰·주차권 + 가맹점/충전소 지도
```

## 모듈 분할 (Claude Code 지시 단위)

### Module 0 — 프로젝트 스캐폴딩
- 모노레포 구조: `/data`(정적 JSON·GeoJSON), `/pipeline`(Python 전처리·예측·최적화), `/web`(대시보드), `/app`(꿀차지 UI)
- 웹: React + Vite. 지도 Leaflet, 차트 Recharts. 서버리스(정적 호스팅 가능해야 함)
- 앱: 데모 목적이므로 React 모바일 웹뷰로 구현 (네이티브 불필요)

### Module 1 — 데이터 파이프라인 (`/pipeline`)
- 입력: KPX 출력제어량 CSV, 기상 CSV, 수요 CSV
- 처리: 결측 보간, 시간 정렬, 피처 엔지니어링(시간대·요일·계절, 일사량·풍속 lag)
- 출력: 학습용 통합 데이터프레임 → parquet

### Module 2 — 예측 모델 (`/pipeline/forecast`)
- LightGBM·XGBoost 각각 학습: 발전량, 수요, 차량 유휴시간
- 시계열 분할 검증, MAPE·R² 비교 테이블 출력 (→ 발표용 검증 슬라이드 근거)
- 출력: 예측 시계열 JSON + 성능 지표 JSON

### Module 3 — MILP 최적화 (`/pipeline/optimize`)
- 라이브러리: OR-Tools 또는 PuLP
- 결정변수: 시간별·차량군별 충/방전량
- 제약: 다음 운행 전 목표 충전량 보장, 충전기 용량, SoC 범위
- 목적함수: 출력제어 회피 편익 − 배터리 열화비용
- EV 대수 {100, 300, 500, 1000, ...} 시나리오별 반복 실행 → `scenarios.json`

### Module 4 — 입지 데이터 통합 (`/data`)
- QGIS 산출 GeoJSON을 받아 스키마 검증: 셀별 `surplus_score`, `commerce_score`, `access_score`, `total_score` 컬럼 필수
- 좌표계 EPSG:4326 확인, 웹 로딩용 경량화(simplify)

### Module 5 — 웹 대시보드 (`/web`)
- 5a: 레이아웃 + 입지 지도(W1) — 적합도 색상 스케일(#ffcd4a 계열), 셀 클릭 → 점수 분해 팝업
- 5b: 스케줄 뷰(W2) — 잉여곡선 area + 충/방전 bar 오버레이
- 5c: EV 슬라이더(W3) — scenarios.json 보간, 절감액·회피 MWh·투자 이연액 카드 갱신
- 5d: 시나리오 비교(W4)
- 5e: Claude API 브리핑(W5) — 현재 슬라이더 값·선택 시나리오를 프롬프트에 넣어 "정책 브리핑" 생성. 실패 시 사전 생성 브리핑 fallback 표시(시연 안정성)

### Module 6 — 꿀차지 앱 (`/app`)
- 6a: 홈 — 오늘의 최적 참여시간 + 예상 수익(schedule.json 기반)
- 6b: 수익 대시보드 — 누적 방전량·정산액·기여도
- 6c: 꿀 지갑 — 쿠폰·주차권 목록 + 가맹점/충전소 지도(siting.geojson 재사용)
- 브랜딩: #ffcd4a, "주차하고 꿀 받으세요" 카피, 꿀단지 아이콘

### Module 7 — 시연 안정화
- 모든 API 호출에 fallback(사전 생성 데이터)
- 오프라인에서도 동작하는 빌드 확인
- 데모 시나리오 스크립트: 발표 흐름대로 클릭 순서 고정

## 권장 지시 순서

Module 0 → 1 → 2 → 3 (파이프라인 완성 후 정적 데이터 확보) → 4 → 5a~5e → 6 → 7

파이프라인(1~3)과 프론트(5~6)는 데이터 스키마만 먼저 합의하면 병렬 진행 가능. **가장 먼저 `scenarios.json`·`siting.geojson`·`schedule.json`의 스키마를 확정할 것** — 이것이 팀 전체(임유정·Fatima ↔ 박재성)의 인터페이스 계약이다.
